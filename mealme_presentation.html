The input

Set portions, diets, and a budget on one screen.

The engine

The output

Ten mains for the week, kitchen ready, exported as CSV or Markdown.

What canteen staff do

Four actions. Plan to plate.

Plan a week
Mon to Fri, meat plus vegetarian, every dish costed, calorie counted, allergen flagged.
Reroll one dish
Replace a single day and track. The rest of the week stays put.
Cover diets
Allocate portions to gluten, nut, or dairy free. The menu swaps only what carries the restriction.
Hand to the chef
Browse the catalogue, read week level KPIs, export the plan in one click.

Logical architecture

One day at a time. The model proposes, code does the rest.

Shortlist

Role tag and sample to 46 real products per track

Propose

LLM returns both tracks as JSON for one day

Validate

Cost, kcal, allergens recomputed from the catalogue

Repair

Invalid days retry with the issues as feedback

WeekPlan

JSON, Markdown, CSV, attempt log for the chef

Key design decisions

Five choices. And why.

LLM composes, code verifies
The model returns only names and quantities. Code owns every number.
Per-day calls
Smaller outputs mean less malformed JSON. A bad Wednesday retries alone.
Shortlists over the catalogue
46 real products per track. The model copies IDs it can see.
Days recompute independently
One path handles initial planning and a swap this dish request.
Diets are variants
The same dish, swapping only the ingredient that carries the restriction.
Catalogue self-correction

Goat butter, labelled vegan. The input data has bugs too.

Goat butter is dairy, so it can never be vegan. The vegetarian track admits any vegan row, so the mislabel lands on a vegetarian plate. The same instinct that guards the output guards the input.

Narrow

Only dairy and egg groups still labelled vegan can hide this bug. Code narrows the catalogue to that handful of suspects.

Judge

One LLM call decides plant or animal for each. A keyword list fails here: "goat" trips "oat", and "almond yogurt" is plant.

Patch

Corrections patch the in-memory catalogue at startup, log at boot, and surface at /api/data-quality.

Ingredient shortlisting

The model never sees all 3,137. Code hands it 46.

3,137 products

Available and track legal, mapped from 29 ingredient groups to 6 roles

seeded sample per role carb 8 protein 10 vegetable 12 sauce 8 flavour 8
ROLE_BY_GROUP lookup, no embeddings
46 real IDs per track

Rendered into the prompt as role grouped CSV.

Meat stays meat

On the meat track, protein rows are forced to dietary_class meat.

No hallucination

The model can only copy IDs it can see, and code rechecks each.

How the LLM is prompted

The prompt does the work. A bounded loop replays it.

prompts.py
# SYSTEM, fixed every call
"You are a canteen menu planner."
  only product_ids from the shortlists
  carb + protein + vegetable + sauce
  meat has meat, vegetarian has none
  respond with JSON only, no prose

# USER, grows day by day
Already planned (avoid repeats): ...
Protein groups used, pick another: ...
Both shortlists, as role-grouped CSV

# USER, on retry: validator strings
"Fix these problems:"
  - product_id 4021 does not exist
  - dish has no sauce/condiment
planner.py
# try at most N times, then give up
for attempt in range(max_attempts):
    # ask; feedback is empty on the 1st try
    raw = llm.propose_day(
        day, shortlists, feedback)
    # parse + check against the catalogue
    issues = validate_dish(
        parse(raw), catalogue)

    if not issues:                  # clean
        return enrich(proposal)     # done

    # the errors become the next prompt
    feedback = issues

# out of tries: fail loud, name the day
raise PlanningError(day)

Parse and type

product_id is an int, grams are positive. Malformed output raises, and the loop treats it like any other rejection.

Catalogue rules

Each id exists and is available, track legality both ways, no excluded allergen, all four roles, no duplicate id. A missing id is a caught hallucination.

Numeric bounds

Carb lines to 450 g, others to 250 g, plate 250 to 950 g, cost in budget. Kcal and cost recomputed from the catalogue. Kcal band and a 5% overshoot are warnings.

Issues go back to the model as repair feedback. Warnings pass the dish but surface on the chef's plan.

Evaluation · correctness over coverage

Correctness, measured three ways. One named gap.

By construction
100%
Structural rules hold for every dish. The validator gates the chef, so existence, track, allergens, and the four roles are correct by design.
Failure modes
25
Property tests, one per way to be wrong: hallucinated id, meat on veg, allergen leak, bad grams, missing sauce, malformed JSON. Offline via the mock client.
The honest gap
1
Semantic quality. Four roles present passes beef plus custard plus rice. Taste and coherence go to a self-critique pass and human review.

Enforce every checkable property as a hard invariant, test every failure mode the case names, and hand taste to a human.
